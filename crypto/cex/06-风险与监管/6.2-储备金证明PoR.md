# 6.2 储备金证明（Proof of Reserves, PoR）

> 深度：🟡 → 🔴
> FTX 之后行业最重要的透明化机制。这篇讲它的原理、技术（默克尔树）和局限。

## 为什么需要 PoR

FTX 暴雷暴露了核心问题：**用户根本不知道交易所是否真的持有它所欠的资产。**
PoR 就是试图用密码学手段，让交易所**可验证地证明「我有足够的钱还给所有用户」**。

## PoR 要证明两件事 ⭐

```
偿付能力健康 = 储备金(有多少资产) ≥ 负债(欠用户多少)
                    │                      │
            Proof of Reserves      Proof of Liabilities
            (证明资产)              (证明负债)
```

### 1. Proof of Reserves（资产证明）
- 交易所链上确实持有这么多资产。
- 方法：公布钱包地址，链上可直接验证余额（或用密码学证明对地址的控制权）。

### 2. Proof of Liabilities（负债证明）⭐ 更难
- 交易所**对用户的总负债**（所有用户余额之和）是多少。
- 难点：既要证明总额真实，又不能泄露每个用户的隐私。
- 解法：**默克尔树（Merkle Tree）**。

## 默克尔树 PoR 原理 ⭐

```
                    Root Hash (公开发布)
                   /            \
              Hash(AB)        Hash(CD)
              /     \          /     \
          H(A)     H(B)     H(C)     H(D)
           │        │        │        │
        用户A余额  用户B余额  用户C余额  用户D余额
```

### 怎么工作
1. 把所有用户的余额作为叶子节点，两两哈希，层层向上，最终得到一个**根哈希（Root Hash）**。
2. 交易所**公开根哈希**（代表了全部负债的"指纹"）。
3. **每个用户可以验证**：交易所给你一条「默克尔路径」，你能验证「我的余额确实被计入了这棵树」，且不泄露别人的数据。
4. 配合资产证明，对比「资产根 ≥ 负债根代表的总额」。

### 妙处
- 用户能验证自己被算进了总负债（交易所没有偷偷少算负债来虚报偿付率）。
- 同时保护隐私（你只看到自己的，看不到别人的）。

## PoR 的局限 ⚠️（必须知道）

PoR 不是万能的，有几个致命局限：

| 局限 | 说明 |
| --- | --- |
| **只是某一时刻快照** | 审计当天达标，第二天就能挪走 |
| **借币应付审计** | 审计前借入资产撑场面，审计后归还(window dressing) |
| **隐藏负债** | 链下债务、表外负债看不到，可能漏报负债虚高偿付率 |
| **资产真实性** | 用自家发行的币(如 FTT)充数，价值虚高(FTX 教训) |
| **控制权证明** | 公布的地址不一定真由交易所独立控制(可能借来的) |
| **负数余额/伪造** | 早期默克尔树方案可被用负余额作弊(后改进) |

> ⚠️ **关键结论**：PoR 是进步，但**不能完全替代第三方审计**。
> 一个负责任的交易所应该：**PoR（链上可验证）+ 独立第三方审计 + 持续披露**三管齐下。
> 只发 PoR 而无审计，仍可能造假。

## PoR 的演进

- **简单版**：只公布钱包地址证明资产（不证明负债，不完整）。
- **默克尔树版**：证明负债且保护隐私（主流，币安、OKX 等采用）。
- **ZK（零知识证明）版**：用零知识证明做更严谨、隐私更好的偿付能力证明（前沿方向）。

## 怎么看一个交易所的 PoR

- 是否同时有**资产证明 + 负债证明**（只有资产证明不够）。
- 是否**定期更新**（一次性的没意义）。
- 是否有**第三方审计**配合。
- 储备率是否 ≥ 100%（足额储备）。
- 资产构成是否健康（是不是大量自家平台币充数？FTX 式风险）。

## 小结

- PoR 要同时证明「资产」和「负债」，核心是 `储备 ≥ 负债`。
- 负债证明用**默克尔树**：用户可验证自己被计入总负债，同时保护隐私。
- 局限明显：快照、借币应付、隐藏负债、自家币充数——**不能替代第三方审计**。
- 看 PoR 要看：资产+负债都证明、定期更新、有审计、足额、资产健康。
