# 8.4 MEV 实战剖析

> 深度：🔴🔴 极硬核
> 把夹击攻击的数学、搜索者的流程、PBS 的细节讲透。前置：[5.2 MEV 详解](../05-市场与博弈/5.2-MEV详解.md)、[8.1 AMM 数学](./8.1-AMM数学全解.md)。

> ⚠️ 本文剖析 MEV 提取的原理，目的是让你理解和防御，不是教你攻击。

## 一、夹击攻击的数学 ⭐

受害者要在 `x*y=k` 池里用 USDC 买 ETH，设了滑点容忍 `s`。搜索者这样夹：

```
初始池: (x_eth, y_usdc), 即时价 P = y/x
受害者交易: 投入 V usdc 买 ETH, 能接受的最差价对应 minOut(滑点 s)

搜索者三步(在同一区块内, 顺序由它控制):
┌─ 前置(frontrun): 搜索者先投入 A usdc 买 ETH
│     → 推高池价, 池变成 (x', y')
│  受害者交易: 在被推高的池子里成交, 拿到比预期少的 ETH(刚好卡在滑点上限)
└─ 后置(backrun): 搜索者把刚买的 ETH 全卖回
      → 因为受害者的买单又推高了价, 搜索者卖在更高价 → 获利
```

### 搜索者的最优夹击量
```
搜索者求解: 选择前置量 A, 最大化利润
约束: 受害者交易后的成交价 ≤ 受害者的 minOut(否则受害者交易回滚, 夹击失败)
→ 受害者设的滑点 s 越大 → 搜索者可塞入的 A 越大 → 夹击利润越高
→ 这就是"滑点设太高被夹更狠"的数学原因(见 5.2)
```

> ⭐ **关键洞察**：夹击利润直接由「受害者的滑点容忍度」决定。滑点是受害者愿意承受的最大价格恶化，搜索者就把它**吃满**。所以**合理的最小滑点**是最简单的防御。

## 二、搜索者（Searcher）的工作流

```
1. 监听内存池(mempool): 实时扫描待打包的交易, 找有利可图的(大额 swap、可清算仓位、套利机会)
2. 模拟(simulate): 在本地 fork 的链状态上模拟"如果我插入这些交易会赚多少"
3. 构造 bundle: 把 [前置, 受害者交易, 后置] 打包成一个原子序列
4. 竞价: 通过 Flashbots 等把 bundle 提交给区块构建者, 出价(贿赂/小费)争取被打包
5. 上链: 构建者选利润最高的方案打包出块, 搜索者获利
```

- 关键能力：**速度**（抢先发现）、**模拟精度**（算准利润）、**出价策略**（既要赢得排序又要留利润）。

## 三、PBS 架构详解（以太坊如何组织 MEV）⭐

```
       搜索者们                  构建者们                 提议者(验证者)
  ┌──────────────┐         ┌──────────────┐         ┌──────────────┐
  │ Searcher A   │ bundle  │              │  区块+出价 │              │
  │ Searcher B   │────────▶│  Builder     │─────────▶│  Proposer    │
  │ Searcher C   │         │ (组装最优区块) │  (via      │ (选出价最高   │
  └──────────────┘         └──────────────┘  relay)   │  的区块上链)  │
       找机会、组bundle          打包成完整区块            └──────────────┘
                                竞争谁的区块价值最高
```

### 角色分工
| 角色 | 职责 |
| --- | --- |
| **Searcher（搜索者）** | 找 MEV 机会，组装交易 bundle，出价 |
| **Builder（构建者）** | 收集 bundle + 普通交易，组装成「价值最大化」的完整区块 |
| **Relay（中继）** | 在构建者和提议者间传递，防作恶（提议者看不到内容直到承诺） |
| **Proposer（提议者/验证者）** | 选出价最高的区块头签名出块，赚 MEV 收益 |

### 为什么要 PBS（分离）💡
```
不分离的问题: 如果验证者自己又找 MEV 又出块 → 大验证者优势碾压 → 加剧中心化
PBS 的目的:   把"找 MEV/构建区块"(复杂、专业)和"出块"(简单)分开
            → 普通验证者不用会找 MEV, 也能通过拍卖拿到 MEV 收益
            → 民主化 MEV 收益, 缓解中心化
```

- **MEV-Boost**：Flashbots 实现的 PBS 中间件，让验证者接入构建者市场。

## 四、MEV 的分类（按危害）

```
"有害" MEV(直接损害用户):
  - 夹击/抢跑 → 散户被抽税
"中性/有益" MEV(维持系统运转):
  - 套利 → 校准价格(但利润来自 LP)
  - 清算 → 及时清算坏账(协议需要)
  - 回填(backrun) → 不前置, 只在后面跟着套利, 不伤用户
```

- 趋势：用机制（私有交易、批量拍卖、MEV 再分配）**抑制有害 MEV，保留有益 MEV**。

## 五、防御 MEV 的技术（汇总）⭐

```
用户侧:
  - 合理最小滑点(别给夹击空间)
  - 私有交易通道(Flashbots Protect/MEV Blocker): 交易不进公开 mempool
  - 抗 MEV 聚合器(CoW: 批量拍卖+需求巧合, 见 2.4)
协议/机制侧:
  - 批量拍卖(同一区块内统一清算价, 消除排序优势)
  - 加密内存池(交易加密, 打包后才解密 → 看不到内容就没法夹)
  - 公平排序(FCFS/阈值加密等研究方向)
  - MEV 再分配(把 MEV 收益返还给用户/应用, 如 MEV-Share)
```

## 六、对照 CEX ⭐

> [CEX 8.6 低延迟](../../cex/08-深水区/8.6-低延迟与性能工程.md) 里，速度优势让做市商/套利者占优，但订单不公开，「抢跑」属违规黑幕。
>
> DEX 的 MEV 是**把「排序权」明码标价、公开拍卖**——
> - CEX：速度竞争在私有系统内，抢跑是作弊。
> - DEX：排序竞争在公开市场（PBS 拍卖），是机制的一部分。
>
> 又一次印证：**「信息/排序优势的获利」在两个赛道都存在，只是 CEX 藏在黑盒、DEX 摆上明牌。** 这是贯穿两个赛道的深层主题（见 [DEX 7.2 收尾](../07-未来发展/7.2-CEX-DEX融合与终局.md)）。

## 小结

- 夹击数学：搜索者前置推高价、让受害者卡在滑点上限成交、后置卖出获利；利润由受害者滑点容忍度决定（滑点越高被夹越狠）。
- 搜索者流程：监听内存池→模拟→组 bundle→竞价→上链。
- PBS（搜索者/构建者/中继/提议者分离）民主化 MEV 收益、缓解中心化；MEV-Boost 是其实现。
- MEV 分有害（夹击/抢跑）和有益（套利/清算）；趋势是抑害留益。
- 防御：合理滑点、私有通道、抗 MEV 聚合器、批量拍卖、加密内存池、MEV 再分配。
