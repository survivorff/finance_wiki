# 3.4 账户与清结算系统

> 深度：🔴 硬核
> 这是交易所的「总账」，管着每个用户欠多少、有多少。一分钱都不能错。

## 核心理念：用户余额是数据库里的一行记录 ⭐

- 用户 A 在所内转 1 BTC 给 B，**不上链**，只是数据库里 A 余额 -1、B 余额 +1。
- 这是 CEX 高性能的根源，也是它和区块链「最终性」的本质差别。
- 用户看到的「我有 0.5 BTC」，本质是「交易所数据库记录：交易所欠你 0.5 BTC」。

## 一、复式记账（Double-Entry Bookkeeping）⭐

交易所的账户系统必须遵守会计学的复式记账原则。

### 核心：每一笔资金变动都有「借」「贷」两方，全局恒等
```
任何时刻必须成立:
所有用户余额之和 + 平台各类账户 = 平台实际持有/应持有的总资产
```

### 为什么必须复式记账
- **账目永远可平衡、可审计**：任何一笔变动都能追溯来源去向。
- **不会凭空产生或消失资金**：一方加必有另一方减。
- 这是**储备金证明（PoR）和审计的基础**——没有严谨的复式记账，根本没法证明偿付能力。

### 例子：用户用 100 USDT 买 0.001 BTC（手续费 0.1%）
```
用户USDT账户:  -100 USDT
用户BTC账户:   +0.000999 BTC
平台手续费账户: +0.000001 BTC
对手方(卖家):  对应反向变动
———————————————————————————
全局资产守恒，借贷平衡 ✓
```

## 二、一致性要求：原子性事务 ⭐

资金变动必须是**原子操作**——要么全部成功，要么全部失败，绝不能出现中间态。

> ⚠️ **反例（绝对不能发生）**：扣了用户的 USDT，但 BTC 没加上 → 用户凭空损失。
> 或者反过来 → 平台凭空亏损。这种 bug 在金融系统里是「P0 级灾难」。

实现手段：
- **数据库事务（ACID）**：单库内用事务保证原子性。
- **分布式事务**：跨服务/跨库时用 TCC、Saga、消息事务等模式。
- **幂等性**：同一操作重复执行结果一致（防止网络重试导致重复扣款/加款）。

## 三、高并发下的一致性挑战

热门账户（如平台手续费账户、热门交易对）会被高频并发修改，怎么保证不出错？

| 手段 | 说明 |
| --- | --- |
| **乐观锁/版本号** | 更新时校验版本，冲突则重试 |
| **账户分片** | 把单账户拆成多个子账户分散热点 |
| **串行化** | 同一账户的操作串行处理（类似撮合的单线程思想） |
| **预冻结机制** | 下单时先冻结资金，成交后再实际扣减（避免超卖） |

### 预冻结（Freeze）机制详解
```
1. 用户下买单(100 USDT) → 冻结 100 USDT(可用余额-100，冻结余额+100)
2. 订单进入撮合
3a. 成交 → 扣减冻结的 USDT，增加 BTC
3b. 撤单 → 解冻 USDT(冻结余额-100，可用余额+100)
```
> 冻结机制保证用户不能用同一笔钱重复下单（双花的内部版本）。

## 四、账户体系的层次

一个用户在交易所往往有**多个子账户**：
```
用户
├── 现货账户(spot)
├── 合约账户(U本位/币本位)
├── 杠杆账户(margin)
├── 理财账户(earn)
└── 资金账户(funding，充提中转)
```
- 各账户独立记账，资金在账户间「划转」。
- 这种隔离便于风险隔离（合约爆仓不会动你现货账户的钱，除非全仓模式）。

## 五、对账（Reconciliation）⭐

交易所必须持续核对「账面」和「实际」：

```
内部账目(数据库所有用户余额之和) ?= 链上实际持有(钱包真实资产)
```
- **链上对账**：定期核对数据库记账总额 vs 钱包链上余额。
- **差异即告警**：对不平 = 可能有 bug、被盗、或内部作恶，必须立即排查。
- 这是发现异常的重要防线，也是 PoR 的数据来源。

## 六、和传统金融的关系 💡

> 复式记账、清结算分离、原子事务、对账——**这些全都是从传统银行核心系统继承来的**。
> 加密交易所在账务层面，和一家银行的核心系统没有本质区别。
> 不同的是：传统银行有央行、有存款保险、有强监管做最后兜底；
> 而 CEX 的账户系统**只有自己对自己负责**（FTX 就是在这层造假/挪用）。

## 小结

- 用户余额本质是数据库记录（交易所的负债），所内交易只改数字不上链。
- 必须用**复式记账**保证账目恒平、可审计，这是 PoR 的基础。
- 资金变动必须**原子 + 幂等**，避免凭空增减资金。
- 高并发靠乐观锁/分片/串行化/预冻结解决。
- **对账**（账面 vs 链上实际）是发现异常的关键防线。
