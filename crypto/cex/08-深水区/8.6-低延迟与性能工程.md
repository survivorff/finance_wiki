# 8.6 低延迟与性能工程

> 深度：🔴🔴 极硬核
> 撮合引擎要做到微秒级延迟，靠的是一整套「机制同情（mechanical sympathy）」的工程手艺。
> 前置：[3.2 撮合引擎](../03-技术架构/3.2-撮合引擎.md)、[8.1 撮合引擎实现](./8.1-撮合引擎实现细节.md)。

## 一、延迟从哪来（敌人画像）

要做低延迟，先认清延迟的来源，按数量级排：

```
操作                          典型延迟量级
───────────────────────────────────────
L1 缓存访问                   ~1 ns
L2 缓存                       ~4 ns
主内存访问                    ~100 ns
锁竞争/上下文切换              ~1-10 μs
系统调用                      ~1 μs
GC 停顿(Java)                ~ms 级(灾难!)
磁盘 I/O                      ~ms 级
网络往返(同机房)              ~μs-ms
网络往返(跨区)                ~ms-数十 ms
```

> 💡 **核心矛盾**：撮合要微秒级，而磁盘/网络/GC 是毫秒级——差了 1000 倍。
> 所以撮合的一切设计，都是为了**待在内存和 CPU 缓存里，远离磁盘、锁、GC、系统调用**。

## 二、机制同情（Mechanical Sympathy）⭐

> 概念来自赛车手 Jackie Stewart：「要开好车，得理解车的机械原理。」
> 写高性能系统，要理解 CPU/内存/缓存的工作机制，顺着硬件的脾气写代码。

### 1. 缓存友好（Cache-Friendly）
```
CPU 按"缓存行(cache line, 通常64字节)"加载内存
→ 连续内存访问(数组) 远快于 随机访问(链表跳指针)
→ 数据结构尽量用数组/连续内存, 提升缓存命中率
```

### 2. 避免伪共享（False Sharing）
```
两个线程频繁写的变量若落在【同一缓存行】
→ 缓存行在核间反复失效/同步, 性能暴跌
解法: 缓存行填充(padding), 让热点变量独占缓存行
```

### 3. 无锁与单写者原则
```
锁 = 竞争 + 上下文切换 + 不确定延迟
LMAX Disruptor 的核心: 单写者(single writer) + 环形缓冲 + 内存屏障
→ 避免锁, 用 CAS/内存屏障做同步
撮合引擎单线程, 本身就规避了大部分锁问题
```

## 三、避免 GC（Java 系尤其关键）

GC 停顿是低延迟的头号杀手（一次 Full GC 可能停几十 ms）。

```
策略:
- 对象池/复用, 避免运行时大量分配 → 减少 GC 触发
- 堆外内存(off-heap), 自己管理生命周期
- 值类型/原始类型数组, 避免对象装箱
- 选低延迟 GC(ZGC/Shenandoah) 或干脆"零分配"设计
- 预热(warm-up): JIT 编译热点代码后再对外服务
```

> 💡 这也是为什么有人用 C++/Rust 写撮合——**没有 GC，延迟更可控**。
> 但顶级 Java 系统通过「零分配 + 堆外 + 预热」也能做到极低延迟（LMAX 就是 Java）。

## 四、内核旁路与硬件（极致场景）

传统玩家用得多，加密所部分采用：

```
- 内核旁路网络(Kernel Bypass): DPDK/Solarflare, 网络包不走内核协议栈, 省系统调用
- 用户态网络栈: 直接在用户态处理网络
- CPU 绑核(affinity) + 隔离核(isolcpus): 撮合线程独占一个核, 关掉中断
- NUMA 感知: 内存分配在本地 NUMA 节点, 避免跨节点访问
- 关闭超线程/降频/C-states: 追求确定性延迟
- FPGA: 极端场景用硬件做行情解析/风控前置(传统 HFT 常见)
```

## 五、时间与确定性

### 时间戳的处理 ⭐
```
撮合逻辑里【不能直接调 now()】(不确定 + 系统调用开销)
→ 时间由【定序器】在入口统一赋予, 并记入日志
→ 撮合引擎读的是"日志里的时间", 保证重放确定性(见 8.1)
```

### 抖动（Jitter）比平均延迟更重要
```
金融系统关心的不只是"平均延迟", 而是"延迟分布的长尾"(p99/p99.9)
→ 一次 10ms 的卡顿(尽管平均很低)可能就错过行情/造成不公平
→ 工程目标: 压低 p99.9, 追求"确定性的低延迟"而非"偶尔很快"
```

## 六、可扩展性 vs 低延迟的张力

```
撮合引擎本身: 单线程不可水平扩展(为了一致性) → 靠"单核足够快"
扩展手段:    按【交易对】分片, 不同交易对跑在不同引擎实例
            (BTC/USDT 一个引擎, ETH/USDT 另一个, 互不影响)
            → 整体吞吐随交易对数水平扩展, 单对内仍单线程保序
```

> 💡 这是个漂亮的设计：**单对内"单线程保一致"，跨对间"分片求扩展"**。
> 既要金融级一致，又要互联网级吞吐，靠的就是这种「按业务键分片」。

## 七、性能工程的优先级（务实建议）

```
1. 先做对(正确性) → 撮合错了再快也没用
2. 架构层优化(单线程/内存/分片/异步) → 收益最大
3. 数据结构优化(缓存友好/无锁) → 收益大
4. JVM/GC 调优 或 换 C++/Rust → 收益中
5. 内核旁路/FPGA → 边际收益, 复杂度高, 按需
```

> ⚠️ 不要过早优化。先用清晰正确的架构(单线程内存撮合+分片)拿到 90% 的性能，
> 再针对 profiling 出的真实瓶颈做底层优化。盲目上 DPDK/FPGA 是工程灾难。

## 小结

- 延迟敌人按量级：缓存(ns) < 内存(100ns) < 锁/系统调用(μs) < GC/磁盘/网络(ms)。撮合要待在内存和缓存里。
- 机制同情：缓存友好(数组)、避免伪共享(padding)、无锁单写者(Disruptor)。
- 避免 GC（对象池/堆外/零分配）是 Java 低延迟关键，也是有人选 C++/Rust 的原因。
- 极致场景用内核旁路、绑核、NUMA、FPGA；时间由定序器统一赋予保确定性。
- 关注 p99.9 长尾抖动而非平均；按交易对分片实现「单对一致 + 跨对扩展」。
