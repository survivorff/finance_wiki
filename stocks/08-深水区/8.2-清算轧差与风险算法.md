# 8.2 清算轧差与风险算法

> 深度：🔴🔴 极硬核
> 清算所如何用轧差压缩交收量、如何用保证金模型管理风险。这是 [CEX 风控](../../crypto/cex/08-深水区/8.2-永续合约数学全解.md) 的"传统金融原版"。
> 前置：[04 清算交收模块](../04-清算交收/README.md)。

## 一、多边轧差（Multilateral Netting）⭐

[4.1](../04-清算交收/4.1-清算与交收概念.md) 讲了轧差的概念，这里讲算法。

### 双边轧差
```
A 对 B: A 买 B 卖 100 股, 同时另一笔 A 卖 B 买 30 股
→ 净额: A 净买 70 股(对 B)
```

### 多边轧差（CCP 的核心）⭐
```
有了 CCP(见 4.2), 所有人都对 CCP 轧差:
  会员 A: 对 CCP 净买 70 股
  会员 B: 对 CCP 净卖 50 股
  会员 C: 对 CCP 净卖 20 股
→ CCP 只需处理每个会员的"净头寸", 而非所有双边关系
→ 资金/证券交收量大幅压缩(可达 90%+)
```

```
轧差计算(简化):
对每个会员 m, 每个证券 s:
  net_position[m][s] = Σ(m 买入 s) − Σ(m 卖出 s)
对每个会员 m:
  net_cash[m] = Σ(m 卖出金额) − Σ(m 买入金额)
→ 交收时, 每个会员只需收付净证券和净现金
```

- 💡 **多边轧差是 CCP 存在的核心价值之一**：N×N 的双边关系压缩成 N 个对 CCP 的净头寸。
- 对照 [CEX 内部记账](../../crypto/cex/03-技术架构/3.4-账户与清结算.md)：CEX 也是「内部轧差」（用户间交易只改净余额），原理一样，但 CEX 是单一主体内部做，CCP 是跨多个独立会员做。

## 二、保证金模型（Margin Models）⭐

CCP 要向会员收保证金，覆盖「会员违约 + 平仓期间价格波动」的风险。怎么算保证金？

### 初始保证金的核心问题
```
保证金要覆盖: "从会员违约 到 CCP 平掉其头寸"期间的潜在损失
关键: 估计头寸在平仓窗口内的"最大可能损失"
→ 这就是风险价值(VaR)和压力测试的用武之地
```

### 方法一：SPAN（情景模拟法）
```
SPAN(Standard Portfolio Analysis of Risk): 期货/期权清算的经典模型
做法:
  1. 设定一组"价格 + 波动率"变动情景(如涨跌 10%、波动率变化的组合)
  2. 计算投资组合在每个情景下的盈亏
  3. 取"最坏情景的损失"作为保证金要求
→ 基于情景的组合风险评估
```

### 方法二：VaR / 历史模拟
```
VaR(Value at Risk, 风险价值):
  "在 99% 置信度下, 未来 N 天最大可能损失是多少"
历史模拟 VaR: 用历史价格变动, 模拟当前组合的损失分布, 取尾部分位数
→ 现代 CCP 多用 VaR 类模型(如 Expected Shortfall)
```

- 💡 **对照 [CEX 合约保证金](../../crypto/cex/08-深水区/8.2-永续合约数学全解.md)**：
  - CEX 用「维持保证金率 + 标记价 + 强平」——相对简单粗暴（固定比率分档）。
  - CCP 用「SPAN/VaR 组合风险模型」——更精细（考虑整个组合的相关性和情景）。
  - 但本质都是：**预估"违约后平仓期间的最大损失"，要求预存足够保证金覆盖它。**

### 变动保证金（每日盯市）
```
每日盯市(Mark-to-Market):
  每天根据收盘价重算每个头寸的盈亏
  浮亏的一方要追加"变动保证金"(把亏损实时补上)
→ 防止亏损累积到无法覆盖
→ 和 CEX 合约的"实时盈亏 + 追加保证金"完全同理(见 CEX 8.2)
```

## 三、违约瀑布的量化（接 [4.2](../04-清算交收/4.2-中央对手方CCP.md)）

```
某会员违约, CCP 的损失吸收顺序(量化):
  L = 违约会员的平仓损失
  1. 违约方初始保证金 IM_default     → 先扣这个
  2. 违约方违约基金贡献 DF_default
  3. CCP 自有资金(skin in the game) SITG
  4. 其他会员违约基金 Σ DF_others    → 共同分担剩余
  5. 追加征收(assessment) / 应急工具
只要 IM + DF 足够覆盖正常波动, 极端才动用后面层级
→ 保证金定得越保守(覆盖越极端情景), 动用共担的概率越低, 但会员成本越高
```

- 💡 这和 [CEX 的「保险基金 → ADL」](../../crypto/cex/03-技术架构/3.5-清算与风控.md) 是同构的分层兜底，只是 CCP 多了「会员共担」这一层（因为 CCP 是会员共有的基础设施）。

## 四、压力测试与逆向压力测试

```
CCP 定期做:
  压力测试: 在极端但可能的情景下, 保证金 + 违约基金够不够?
  逆向压力测试: "什么情景会击穿我的所有防线?" → 找系统脆弱点
→ 2008 危机后, 监管对 CCP 压力测试要求极严(CCP 是系统重要节点, 见 4.2)
```

## 小结

- 多边轧差把 N×N 双边关系压缩成 N 个对 CCP 的净头寸，大幅减少交收量——CCP 核心价值。
- 保证金模型（SPAN 情景法、VaR 历史模拟）估算「违约后平仓期最大损失」，比 CEX 的固定比率更精细。
- 每日盯市 + 变动保证金实时补亏，和 CEX 合约同理。
- 违约瀑布量化（保证金→违约基金→自有→会员共担）是 CEX 保险基金+ADL 的同构升级版。
- CCP 受严格压力测试要求，因为它是系统重要性节点。
