# 8.4 钱包系统深挖

> 深度：🔴🔴 极硬核
> 把托管钱包的底层拆开：HD 钱包派生、地址管理、MPC-TSS 原理、归集与 Gas 工程。
> 前置：[3.3 钱包与充提](../03-技术架构/3.3-钱包与充提.md)。

## 一、HD 钱包与地址派生（充值地址从哪来）

交易所要给海量用户分配充值地址。不可能为每个用户单独生成并保管一把私钥——用 **HD 钱包（分层确定性钱包，BIP-32/39/44）**。

### 核心思想
```
一个主种子(seed) → 派生出无限多个子密钥/地址
派生路径: m / purpose' / coin_type' / account' / change / address_index
          例: m/44'/0'/0'/0/12345   (第12345个地址)
```

- 用**一个种子**就能确定性地派生出**所有用户地址**，只需保管好种子（+ 派生索引）。
- 给新用户分配地址 = 递增 `address_index` 派生一个新地址，记录 `index ↔ user_id` 映射。

### xpub 派生（热端不放私钥的技巧）💡
```
用"扩展公钥(xpub)"可以在【不接触私钥】的情况下派生所有【公钥/地址】
→ 在线系统只用 xpub 派生收款地址(安全)
→ 私钥(xpriv)留在离线签名端
```
> 这让「在线生成地址」和「离线掌管私钥」分离——在线端被黑也拿不到私钥。

### 账户型链的不同（ETH 等）
- BTC 这类 UTXO 链天然适合「每用户一地址」。
- ETH 这类账户链，很多交易所用**统一充值地址 + 内部归集**，或用 **memo/tag** 区分（如 EOS、XRP、部分交易所的统一地址方案）。

## 二、充值检测的工程细节

```
1. 节点/索引器扫描新区块
2. 解析每笔 tx 的输出地址, 匹配"我们的地址集合"
3. 命中 → 记录待确认充值
4. 跟踪确认数, 达到阈值 → 触发入账(走 8.3 的记账流程, 幂等!)
5. 处理 reorg: 若已确认的区块被回滚, 要能撤销入账
```

- ⚠️ **reorg 处理**：确认数就是为对抗 reorg。但若发生超过确认数的深度重组（极罕见），要有补偿机制。
- ⚠️ **幂等**：同一笔链上充值只能入账一次（用 txhash+vout 做幂等键）。
- **假充值攻击防范**：要校验交易真实成功（如 ETH 要查 receipt status、防 ERC20 转账失败但事件伪造、防 0 转账钓鱼）。

## 三、MPC / TSS 原理（现代托管的核心）⭐

### 多签 vs MPC-TSS 的本质区别
```
多签(Multisig):  链上有 M-of-N 逻辑, 需要 M 个【独立签名】, 链上可见是多签
                 (依赖具体链支持多签脚本/合约)

MPC-TSS(门限签名): 密码学层面把【一把私钥的签名能力】拆成 N 份
                  M 方协同算出【一个普通签名】, 私钥从未完整出现
                  链上看就是普通单签(省 gas、隐私好、跨链通用)
```

### TSS（Threshold Signature Scheme）干了什么
```
密钥生成(DKG): N 方各自持有一个"私钥分片", 没有任何一方有完整私钥
              但它们对应同一个公钥/地址
签名:         M 方(门限)用各自分片参与多轮交互计算
              → 输出一个对该公钥有效的标准签名
              → 全程完整私钥从不在任何一台机器上重组出现
```

> 💡 **为什么这是革命性的**：
> - 没有「完整私钥」这个单点攻击目标（黑掉任何 M-1 台机器都拿不到私钥）。
> - 链上无感知（不像多签暴露策略、也不受链是否支持多签限制）。
> - 可灵活调整门限和分片（员工离职、密钥轮换）。
> 这就是 Fireblocks/Copper 等机构托管的技术核心。

### 攻击面转移（Bybit 教训的技术解释）⭐
即便有 MPC/多签，攻击者转向**「让签名者签下恶意交易」**：
```
攻击者篡改签名者看到的"待签内容"(界面/前端被攻破)
→ 签名者以为在签正常交易, 实际签了"转给黑客"的交易
→ MPC 忠实地为这笔恶意交易产生了有效签名
```
防御：
- **签名前在独立可信设备上验证交易内容**（防界面欺骗）。
- **策略引擎**：白名单地址、金额限制、异常拦截，在签名层强制执行。
- **杜绝盲签**：签名者必须能看清并理解所签内容。

## 四、归集（Sweeping）与 Gas 工程

### 归集的难题
用户充值分散在成千上万个地址，要归集到主钱包。但归集本身要花 Gas：

```
ETH/ERC20 归集的鸡生蛋问题:
  用户充了 USDT(ERC20)到派生地址, 但该地址没有 ETH 付 Gas
  → 无法直接转出 USDT
  解法: 先从"加油站(gas station)"地址打一点 ETH 过去当 Gas
       再发起 USDT 归集转账
       (或用 EIP-4337 账户抽象 / 代付方案)
```

### Gas 优化策略
- **批量归集**：攒一批一起转，摊薄固定成本。
- **择时**：Gas 低谷期（链不拥堵时）归集。
- **UTXO 合并**：BTC 类链定期合并碎片 UTXO，降低未来交易体积。
- **费率预测**：动态估 Gas，避免卡住或多付。

## 五、提币的签名工作流（生产级）

```
用户提币请求
   │
   ▼
风控/合规审核(额度/白名单/AML/冷静期)
   │
   ▼
构造交易(选热钱包UTXO/nonce, 估Gas)
   │
   ▼
策略引擎校验(目的地白名单? 金额阈值? 频率?)
   │
   ▼
MPC/多签 签名(多方/多设备, 签前验证内容)
   │
   ▼
广播上链 → 跟踪确认 → 更新提币状态(幂等记账)
   │
   └── 热钱包不足 → 触发冷钱包补水(更高审批等级)
```

## 六、冷热分层的资金调度

```
        提币需求预测(基于历史流量)
              │
   ┌──────────┴──────────┐
   ▼                     ▼
热钱包水位管理         冷钱包补水策略
(维持 N 天周转量)     (低于阈值→多签从冷转热)
   │                     │
   └── 过多 → 定期归集回冷钱包(降低热钱包风险敞口)
```

> 💡 这是一个「**库存管理**」问题：热钱包放多了风险大，放少了提币卡顿。
> 要在「安全（少放热钱包）」和「体验（提币快）」之间动态平衡，类似零售的库存周转。

## 小结

- HD 钱包用一个种子确定性派生海量地址；用 xpub 在线派生地址而私钥离线。
- 充值检测要处理确认数、reorg、幂等、假充值攻击。
- MPC-TSS 把单私钥签名能力分片，私钥从不重组，是现代托管核心；但攻击转向「欺骗签名内容」（Bybit 教训），需签前验证+策略引擎+杜绝盲签。
- 归集面临 Gas 鸡生蛋问题（加油站方案），需批量/择时优化。
- 冷热分层本质是「安全 vs 提币体验」的库存管理问题。
