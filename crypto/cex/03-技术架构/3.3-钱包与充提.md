# 3.3 钱包与充提系统

> 深度：🔴 硬核
> 这是交易所安全的命门。资产安全 90% 取决于钱包架构。Mt.Gox 和 Bybit 都是栽在这里。

## 核心理念：钱包系统是「链上世界与内部记账的接口」

```
链上世界(真实区块链)  ◀──钱包系统──▶  交易所内部记账(数据库余额)
   真的有币，私钥控制         翻译层          只是数字，快速变动
```

- 用户**充币**：真币从链上进交易所钱包 → 系统给数据库余额 +N。
- 用户**提币**：数据库余额 -N → 真币从交易所钱包转到用户链上地址。
- 所内交易：只动数据库，不碰钱包系统。

## 一、冷热钱包分离 ⭐ 安全的基石

| | 热钱包 (Hot) | 冷钱包 (Cold) |
| --- | --- | --- |
| 联网 | 在线 | 离线（物理隔离） |
| 用途 | 应对日常提币周转 | 存储绝大部分资产 |
| 资金占比 | 少（如 5%~15%） | 多（如 85%~95%） |
| 私钥位置 | 在线服务器（高风险） | 离线设备/HSM（高安全） |
| 风险 | 高（联网易被黑） | 低（需物理接触） |

> ⭐ **设计原则**：热钱包只放「够日常提币周转」的少量资金，绝大部分锁在冷钱包。
> **即使热钱包被完全攻破，损失也被限制在一个可控的小比例。** 这是 Mt.Gox 之后的行业铁律。

```
                 用户提币请求
                      │
                      ▼
            ┌──────────────────┐
            │  热钱包(5-15%)     │ ── 够用 ──▶ 直接签名出币
            │  在线、自动签名     │
            └──────────────────┘
                      │ 不够用时
                      ▼ (人工/多签审批"补水")
            ┌──────────────────┐
            │  冷钱包(85-95%)    │ ── 离线签名 ──▶ 补充到热钱包
            │  离线、多签/MPC     │
            └──────────────────┘
```

## 二、私钥安全方案（三种主流）

### 1. 多签（Multisig）
- 动用资金需要 **M-of-N** 把私钥共同签名（如 3/5：5 把钥匙里至少 3 把）。
- 钥匙分散在不同人/不同地点，**避免单点失守**（一把泄露不够动钱）。

### 2. MPC（多方安全计算）⭐ 现代主流
- 私钥**从不完整存在于任何一处**。被分成多个「分片」，签名时多方协同计算，全程不重组出完整私钥。
- 优势：没有「完整私钥」这个攻击目标；链上看就是普通单签交易（省 gas、隐私好）。
- 机构托管商（Fireblocks、Copper）的核心技术。

### 3. HSM（硬件安全模块）
- 私钥存在**防篡改硬件**里，签名在硬件内完成，私钥永不出硬件。
- 常和多签/MPC 配合使用。

> 💡 多签 vs MPC：多签是「链上层面」需要多个签名（智能合约/脚本实现）；
> MPC 是「密码学层面」把单个私钥的签名能力拆分。现代大所多用 MPC 或 MPC+多签组合。

## 三、充币流程（Deposit）

```
1. 系统给用户分配充值地址(每人独立地址 或 共享地址+memo/tag 区分)
2. 钱包服务监听区块链，扫块发现入账交易
3. 等待"足够确认数"(confirmations)
4. 确认充足 → 给用户账户记账 +N
```

### 关键点：为什么要等确认数
- 区块链可能发生**区块重组（reorg）**，刚打包的交易可能被回滚。
- 等若干个确认（如 BTC 等 1-6 个，ETH 等 12+ 个）能极大降低**双花**风险。
- 💡 这就是为什么充币「到账慢」——不是系统卡，是在等区块确认保安全。不同币种确认数不同。

## 四、提币流程（Withdraw）

```
1. 用户发起提币
2. 风控审核: 额度限制、异常行为、是否触发人工/多签审批、提币地址白名单
3. 从热钱包签名 → 广播上链
4. 热钱包余额不足 → 从冷钱包"补水"(归集，需多签审批)
5. 链上确认后，标记提币完成
```

### 提币的安全控制
- **地址白名单**：只能提到预先设置的地址。
- **额度分级**：小额自动、大额人工/多签审批。
- **冷静期**：新增白名单地址后延迟 24 小时才能提（防黑客改地址秒提）。
- **风控拦截**：账号异常（异地登录、改密码后）暂停提币。

## 五、归集（Sweeping）

- 用户充值地址分散，资金零散。系统定期把这些零散资金**归集到主钱包/冷钱包**统一管理。
- 归集要消耗 gas，需要权衡频率和成本。

## 六、多链支持的复杂性 💡

交易所要支持几十上百条链（BTC、ETH、各种 EVM 链、Solana、Tron...），每条链：
- 地址格式、交易结构、确认机制、gas 机制都不同。
- 同一个币可能在多条链上发行（如 USDT 在 ETH/Tron/Solana 上都有）——**用户选错链充提会丢币**。
- ⚠️ **充错网络是用户资产损失的高发场景**（如把 ERC20 的 USDT 提到只支持 TRC20 的地址）。

## 七、惨痛案例：钱包安全失守

- **Mt.Gox（2014）**：长期被盗未察觉，~85 万 BTC 丢失。
- **Bybit（2025）**：冷钱包签名环节被攻击（疑似界面欺骗/供应链攻击），约 15 亿美元级被盗。
- 💡 **教训演进**：早期是「热钱包被黑」，现在攻击者转向**「签名流程/操作环节」**——
  即便冷钱包，如果签名时被欺骗（显示的和实际签的不一致），照样被盗。
  所以现代安全重点是**「签名前的验证」**（盲签是大忌）。

## 小结

- 钱包系统是链上与内部记账的接口，是交易所安全命门。
- 核心架构：**冷热分离**（热钱包少量周转，冷钱包存大头）。
- 私钥安全：多签 / MPC（现代主流）/ HSM。
- 充币等确认数防双花，提币靠白名单+额度分级+冷静期+风控。
- 现代攻击转向「签名流程」，盲签是大忌；多链和充错网络是用户高发风险。
