# 3.2 撮合引擎（Matching Engine）

> 深度：🔴⭐ 全系统最核心、最硬核的部分
> 撮合引擎是交易所的「心脏」。理解它，就理解了金融级高性能系统的精髓。

## 它要做什么

接收订单 → 维护订单簿 → 按规则撮合成交 → 输出成交结果。
要求：**极致低延迟、绝对正确、强一致、可恢复。**

## 一、订单簿（Order Book）数据结构

### 撮合规则：价格优先 + 时间优先（Price-Time Priority）⭐
- 买单按价格**从高到低**排，卖单按价格**从低到高**排。
- 同一价格内，**先到先得**（时间优先）。

```
卖单 asks
  101.0  | 订单D(1.0) 订单E(0.5)        ← 同价位按时间排队
  100.5  | 订单C(1.5)                  ← best ask
────────────────────────────────────
  100.0  | 订单B(2.0)                  ← best bid
   99.5  | 订单A(1.0)
买单 bids
```

### 工程实现
- 每个**价格档位**维护一个**FIFO 队列**（同价订单按时间排）。
- 价格档位之间用**排序结构**组织：
  - 红黑树 / 跳表（通用，O(log n)）
  - 或「数组 + 价格索引」（价格离散化后用数组直接索引，O(1)，更快但占内存）
- 目标：找到最优价、撮合、插入/撤单都尽量 O(1)~O(log n)。

### 撮合逻辑（限价买单为例）
```
来了一个限价买单(价格P, 数量Q):
  while Q > 0 且 存在卖单 且 最优卖价 <= P:
      和最优卖价的队首订单成交 min(Q, 该订单剩余量)
      更新双方剩余量，产生成交记录
  if Q > 0:  把剩余的 Q 作为新挂单插入买盘
```

## 二、为什么是「单线程 + 全内存」⭐⭐

这是初学者最反直觉、也是最精髓的设计。

### 单线程
- **撮合是强顺序逻辑**：订单的处理顺序直接决定谁先成交（时间优先）。多线程并发会破坏顺序、引入竞态，加锁又带来复杂性和延迟。
- 单线程**顺序处理**：天然无锁、无竞态、逻辑简单、易验证正确性。
- 现代 CPU 单核很快，单线程纯内存撮合可达**每秒数十万~数百万笔**。

### 全内存
- 订单簿全部放内存，撮合时**不碰磁盘、不碰数据库、不发网络请求**——这些都是微秒级延迟的杀手。
- 纯内存操作让单笔撮合延迟降到**微秒级**。

> ⭐ **核心洞察**：把撮合建模成一个**确定性状态机**——
> **相同的输入序列 → 必然得到完全相同的输出和状态。** 没有随机、没有外部依赖、没有并发不确定性。
> 这个性质是后面「持久化恢复」和「高可用」的全部基础。

### 经典参考架构：LMAX Disruptor
- 伦敦的 LMAX 交易所开源的高性能队列框架，是金融业单线程高性能架构的标杆。
- 核心思想：环形缓冲区（Ring Buffer）+ 无锁 + 缓存行填充（避免伪共享）+ 单线程业务逻辑。
- 很多加密交易所的撮合引擎借鉴了这套思路。

## 三、内存数据怎么不丢？事件溯源（Event Sourcing）⭐

**问题**：全内存撮合，一旦进程崩溃/断电，订单簿不就没了吗？
**答案**：引擎本身不负责持久化，靠「输入日志 + 快照」恢复。

```
                  ┌─────────────────────────────────┐
所有订单指令 ──▶  │ 持久化日志(WAL/Kafka) │ ──▶ 撮合引擎(内存)
                  └─────────────────────────────────┘
                         │
                  先落盘，再喂给引擎
                         │
   崩溃恢复: 加载最近"快照" + 重放快照之后的所有日志 = 精确恢复到崩溃前
```

- **预写日志（WAL）/ 事件日志**：所有输入指令（下单、撤单）**先持久化，再处理**。
- **快照（Snapshot）**：定期把内存订单簿的完整状态存盘。
- **恢复**：崩溃后 = 最近快照 + 重放之后的日志。因为是确定性状态机，**重放必然得到一模一样的状态**。

> 💡 这就是为什么 [3.1](./3.1-整体架构总览.md) 里强调「先写消息队列再撮合」——日志即真相（log is the source of truth）。

## 四、高可用：状态机复制

单台引擎再快也会宕机，怎么做高可用而不丢一致性？

- **状态机复制（State Machine Replication）**：把**同一串输入指令**喂给多个引擎副本（主 + 多个备）。
- 因为是确定性状态机，所有副本会得到**完全一致的状态**。
- 主节点挂了，备节点立刻顶上，状态一致，无缝切换。

```
            ┌──▶ 主引擎(对外服务)
输入指令流 ──┼──▶ 备引擎1(热备，状态一致)
            └──▶ 备引擎2(热备，状态一致)
   主挂 → 备1 立即接管(状态完全一致)
```

## 五、订单类型的引擎支持

撮合引擎要正确处理各种订单语义：
- 限价（Limit）、市价（Market）、止损/止盈（Stop/TP）
- IOC（立即成交剩余取消）、FOK（全成或全撤）、Post-Only（只挂单）
- 冰山订单（Iceberg，大单拆成小单隐藏真实数量）

## 六、性能优化手段（工程细节）

- **绑核（CPU Affinity）**：撮合线程固定在一个 CPU 核，避免上下文切换。
- **内存预分配 + 对象池**：避免运行时分配内存和 GC 停顿（Java 系尤其重要）。
- **避免伪共享（False Sharing）**：缓存行填充。
- **批处理**：批量处理日志写入和行情推送。
- **NUMA 感知**：内存就近访问。

## ⚠️ 撮合引擎的「红线」

- **绝不能出错**：撮合错误 = 资金错误 = 灾难。所以逻辑极简、充分测试、可回放审计。
- **绝不能不公平**：必须严格价格-时间优先，否则就是「插队」，破坏市场信任。
- 💡 但闭源撮合也带来质疑：用户**无法验证**交易所是否真的公平撮合、是否有内部抢跑。这是 CEX 的透明度软肋（也是 DEX「链上可验证撮合」的卖点）。

## 小结

- 撮合引擎核心 = 订单簿（价格-时间优先）+ 撮合逻辑。
- 设计精髓：**单线程 + 全内存 = 确定性状态机**，换来极致性能和强一致。
- 不丢数据靠**事件溯源**（先持久化日志，崩溃后快照+重放恢复）。
- 高可用靠**状态机复制**（同样输入喂多副本，状态一致，无缝切换）。
- 红线：绝对正确 + 绝对公平；软肋：闭源不可验证。
