# 3.1 整体架构总览

> 深度：🔴 硬核
> 先建立全局工程视图，再深入各子系统。

## CEX 是什么样的系统

用一句话定性：**一个 7×24 不能停、要求强一致、低延迟、高并发、且管着真金白银的金融交易系统。**

这意味着它的工程难度集合了：
- **高频交易系统**的低延迟要求（撮合）
- **银行核心系统**的资金一致性要求（账户）
- **互联网大厂**的高并发要求（行情、API）
- **安全公司**的攻防要求（钱包、风控）

## 典型分层架构

```
┌─────────────────────────────────────────────────────────┐
│ 接入层    CDN / 负载均衡 / DDoS 防护 / WAF                  │
├─────────────────────────────────────────────────────────┤
│ 网关层    API 网关(鉴权/限流/路由) · WebSocket 网关          │
├─────────────────────────────────────────────────────────┤
│ 业务层    账户 · 订单 · 钱包 · 理财 · 合约 · 法币 · KYC ...   │
├─────────────────────────────────────────────────────────┤
│ 核心层    撮合引擎 · 清算结算 · 风控强平 (最关键、最敏感)      │
├─────────────────────────────────────────────────────────┤
│ 数据层    内存数据库 · 关系库 · 消息队列(Kafka) · 缓存(Redis) │
├─────────────────────────────────────────────────────────┤
│ 基础设施  区块链节点 · 监控告警 · 日志 · 配置中心             │
└─────────────────────────────────────────────────────────┘
```

## 关键数据流：一笔现货下单的生命周期

```
1. 用户点"买入"
   │
2. 接入层(DDoS/WAF) → 网关(鉴权、限流、参数校验)
   │
3. 订单服务: 预冻结资金(账户服务) + 风控校验
   │
4. 订单写入消息队列(持久化) ──┐
   │                         │ 先持久化再处理 = 可恢复
5. 撮合引擎消费订单 ◀─────────┘
   │ (内存撮合，微秒级)
6. 产生成交回报 → 推送到 [清算服务] 和 [行情服务]
   │
7. 清算: 改双方余额、扣手续费(账户服务，复式记账)
   │
8. 行情: 更新盘口、生成K线、推送给所有订阅用户(WebSocket)
   │
9. 用户看到"成交"
```

> ⭐ **关键设计点**：第 4 步「先持久化到消息队列，再喂给撮合引擎」。
> 这保证了即使撮合引擎宕机，也能从日志重放恢复——这是金融级可靠性的根基（详见 [3.2](./3.2-撮合引擎.md)）。

## 微服务划分原则

CEX 后端通常是微服务，但有个**例外原则**：

> **核心交易链路（撮合）追求极致性能和一致性，倾向于单体/单线程；
> 外围业务（理财、KYC、活动）追求迭代速度和隔离，倾向于微服务拆分。**

这种「核心收紧、外围放开」的架构哲学，在金融系统里很典型。

## 典型技术栈（行业常见选择）

| 层 | 常见技术 | 说明 |
| --- | --- | --- |
| 撮合引擎 | C++ / Rust / Java(低延迟优化) | 追求低延迟、可控 GC |
| 业务服务 | Go / Java / Rust | 高并发、易维护 |
| 消息队列 | Kafka / RocketMQ | 持久化、事件溯源 |
| 缓存 | Redis | 行情缓存、会话 |
| 数据库 | MySQL/PostgreSQL + 分库分表 | 账户、订单存储 |
| 行情推送 | WebSocket + 自研网关 | 百万连接 |
| 钱包 | 自研 + HSM/MPC(如 Fireblocks) | 私钥安全 |

> 💡 撮合引擎用什么语言有「门派」之分：追求极致低延迟用 C++/Rust；
> Java 系（如借鉴 LMAX Disruptor 架构）通过避免 GC、绑核、内存预分配也能做到极低延迟。

## 性能量级感受

- 顶级撮合引擎：**单核每秒数十万~数百万笔订单**，撮合延迟微秒级。
- 行情推送：**百万级并发 WebSocket 长连接**。
- 整体要求：**99.99%+ 可用性**（金融系统，宕机即事故 + 用户损失 + 声誉打击）。

## 小结

- CEX 是集高频交易、银行核心、互联网高并发、安全攻防于一身的复杂系统。
- 一笔订单经过「网关→风控→持久化→撮合→清算→行情」的完整链路。
- 架构哲学：核心交易链路收紧（性能/一致），外围业务放开（微服务/迭代）。
- 「先持久化再撮合」是金融级可靠性的根基。
