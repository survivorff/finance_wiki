## 一、复式记账的表结构设计

核心思想：**不要直接 UPDATE 余额，而是 APPEND 不可变的分录（entries），余额由分录推导/累加。**

### 三张核心表

```sql
-- 1. 账户表：每个(用户, 币种, 账户类型)一行
accounts(
  account_id PK,
  user_id, asset, account_type,     -- 如 (user1, USDT, spot)
  balance,           -- 可用余额(物化的当前值，可由 entries 重算校验)
  frozen,            -- 冻结余额
  version            -- 乐观锁版本号
)

-- 2. 交易表：一笔业务操作(一定借贷平衡)
transactions(
  tx_id PK,
  type,              -- trade/deposit/withdraw/transfer/fee/funding...
  ref_id,            -- 业务幂等键(订单成交id等), UNIQUE
  created_at
)

-- 3. 分录表：不可变、只追加。一笔 tx 至少两条分录，借贷相等
entries(
  entry_id PK,
  tx_id FK,
  account_id FK,
  delta,             -- 正=贷(加), 负=借(减)
  balance_after,     -- 记账后余额(快照, 便于审计和追溯)
  created_at
)
```

### 不变量（任何时刻必须成立）⭐
```
1. 每个 tx 的所有 entries 的 delta 之和 == 0   (借贷平衡)
2. 每个 account 的 balance == 该账户所有 entries 的 delta 累加
3. 对每个 asset:  Σ 所有用户 balance == 平台为该 asset 实际持有/应持有的量
```

## 二、一笔成交的记账（伪代码）

用户 A 买、用户 B 卖，成交 0.001 BTC @ 60000 USDT，taker 手续费 0.1%：

```python
def settle_trade(trade):
    with db.transaction():                      # 数据库事务保证原子性
        # 幂等：同一成交只记一次
        if transactions.exists(ref_id=trade.id):
            return
        tx = transactions.create(type='trade', ref_id=trade.id)

        # 买方 A: -60 USDT, +0.001 BTC, -手续费
        entries.append(tx, A.usdt_acc, delta=-60.00)
        entries.append(tx, A.btc_acc,  delta=+0.001)
        fee = 0.001 * 0.001                     # taker 费 0.1%
        entries.append(tx, A.btc_acc,  delta=-fee)
        entries.append(tx, platform.fee_btc, delta=+fee)

        # 卖方 B: +60 USDT, -0.001 BTC (B 是 maker, 假设费率更低/为0)
        entries.append(tx, B.btc_acc,  delta=-0.001)
        entries.append(tx, B.usdt_acc, delta=+60.00)

        assert sum_of_all_deltas(tx) == 0       # 借贷平衡校验
        update_balances_optimistic_lock(...)    # 物化余额, 乐观锁防并发
```

> ⚠️ 注意 fee 的处理：手续费也是分录的一部分，从用户账户「借」、向平台手续费账户「贷」，保持全局平衡。

## 三、幂等性（Idempotency）⭐ 防重复记账

分布式系统里，消息会重复投递、请求会重试。**同一笔业务绝不能记两次账。**

```
机制: 每笔业务有唯一的 ref_id (成交id/提币id/划转id)
      transactions.ref_id 加 UNIQUE 约束
      记账前先查 ref_id 是否已存在 → 存在则直接返回(已处理)
```

这样即使「成交消息重发 3 次」，也只记一次账。幂等是金融系统的生命线。

## 四、冻结机制（下单到成交的资金安全）

```
下单:    available -= amount;  frozen += amount     (一个原子事务内)
成交:    frozen -= filled;     扣减实际成交部分
撤单:    frozen -= remaining;  available += remaining (解冻)
```
- 保证「同一笔钱不能重复下单」（内部版的双花防护）。
- 冻结和解冻都要幂等 + 原子。

## 五、并发控制：乐观锁 vs 串行化

热点账户（平台手续费账户、热门做市商账户）会被高频并发改：

```python
# 乐观锁：更新时校验 version
UPDATE accounts SET balance = balance + :delta, version = version + 1
WHERE account_id = :id AND version = :expected_version
# 影响行数 == 0 → 说明被并发改过 → 重试整个操作
```

- **乐观锁**：冲突少时高效；冲突多时重试开销大。
- **账户分片**：把单个热点账户拆成 N 个子账户，分散写入，定期合并。
- **串行化**：对单账户的操作进队列串行处理（类似撮合的单线程思想）。

## 六、对账（Reconciliation）⭐ 发现异常的最后防线

### 内部对账（账账核对）
```
对每个 asset:
  Σ(所有 entries.delta where asset=X) 应该等于 0
  Σ(用户余额) + 平台账户 应该自洽
  物化的 account.balance 应该 == 重算的 entries 累加值
  → 不一致 = 有 bug，立即告警冻结相关操作
```

### 链上对账（账实核对）⭐
```
内部记账总额(数据库 Σ所有用户该币余额)  ?=  钱包链上实际持有(冷+热钱包余额)

差异分析:
  链上 > 内部:  可能有未入账的充值 / 正常(储备充足)
  链上 < 内部:  ⚠️ 危险信号! 可能被盗、挪用、或 bug → 头等告警
```

- 定期（如每日/每小时）跑全量对账。
- 这就是 [6.2 储备金证明](../06-风险与监管/6.2-储备金证明PoR.md) 的数据基础。

## 七、PoR（默克尔树负债证明）的实现

### 构建负债树
```python
def build_liability_tree(users):
    leaves = []
    for u in users:
        # 叶子 = hash(用户标识nonce + 各币种余额)
        leaf = hash(f"{u.nonce}:{u.balances}")
        leaves.append(leaf)
    # 改进版：节点同时携带"子树余额之和"，防止用负余额作弊
    return merkle_build(leaves)   # 自底向上两两哈希到 root
```

### 改进点（防作弊）💡
- **携带子树和**：每个内部节点存 `hash` + `sum(子树余额)`，让审计能验证「父节点的和 = 子节点和之和」，防止插入负余额虚减总负债（早期 PoR 的漏洞）。
- **用户验证**：交易所给用户其叶子的 Merkle 路径，用户重算到 root 比对公布值，确认自己被计入。

### 局限（再强调）
PoR 只证某一时刻、只证链上资产和声明的负债，**挡不住借币应付审计、隐藏链下负债**。必须配第三方审计（见 [6.2](../06-风险与监管/6.2-储备金证明PoR.md)）。

## 八、资金系统的「红线」清单

```
✅ 金额用整数最小单位，绝不用浮点
✅ 余额变动走"追加不可变分录", 不裸 UPDATE
✅ 每笔业务借贷必须平衡(Σdelta=0)
✅ 所有记账操作幂等(ref_id 唯一)
✅ 所有记账操作原子(数据库事务)
✅ 冻结/解冻配套, 防双花
✅ 持续内部对账 + 链上对账, 差异即告警
✅ 全程审计日志, 任何余额变动可追溯
```

## 小结

- 复式记账落地为「accounts + transactions + entries」三表，余额由不可变分录推导。
- 核心不变量：每笔 tx 借贷平衡、余额=分录累加、用户总额=链上持有。
- 幂等（ref_id 唯一）防重复记账，冻结机制防双花，乐观锁/分片/串行化解决并发。
- 内部对账 + 链上对账是发现挪用/被盗/bug 的最后防线，也是 PoR 的数据基础。
- PoR 用携带子树和的默克尔树防作弊，但仍需第三方审计兜底。
