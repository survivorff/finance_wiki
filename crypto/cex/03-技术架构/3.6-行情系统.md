# 3.6 行情系统（Market Data）

> 深度：🔴 硬核
> 行情系统要把撮合产生的数据，实时推送给可能上百万的在线用户。这是典型的「海量并发广播」问题。

## 行情系统要输出什么

| 数据 | 内容 | 更新频率 |
| --- | --- | --- |
| **盘口深度(Order Book)** | 各价位的买卖挂单量 | 极高频（每笔成交/挂撤单） |
| **最新成交(Trades)** | 逐笔成交记录 | 每笔成交 |
| **K线(Candlestick)** | 开高低收 + 成交量 | 按周期(1m/5m/1h...) |
| **24h 统计** | 涨跌幅、最高最低、成交量 | 秒级 |
| **指数价/标记价** | 合约相关公允价 | 秒级 |
| **资金费率** | 永续合约费率 | 周期性 |

## 核心挑战：百万级并发 + 低延迟广播

```
撮合引擎(1个数据源) ──▶ 行情系统 ──▶ 推送给 1,000,000+ 在线用户
                          ↑
              一对多的海量实时广播
```

行情更新极其频繁（牛市热门币每秒成千上万次盘口变化），还要低延迟推给海量用户。这是工程难点。

## 一、推送技术：WebSocket 长连接

- **为什么不用 HTTP 轮询**：轮询延迟高、浪费资源。行情要「服务器主动推」。
- **WebSocket**：建立一次长连接，服务器有更新就主动推给客户端。
- **百万级长连接**的挑战：单机连接数有限，需要大量推送网关 + 负载均衡 + 连接保活。

## 二、增量推送（Delta / Incremental Update）⭐

**问题**：盘口有几百上千个价位，每次变化都推全量太浪费带宽。
**解法**：先推一份**全量快照**，之后只推**变化的部分（增量）**，客户端本地合并。

```
1. 客户端订阅 → 服务器推一份完整盘口快照(snapshot)
2. 之后只推变化: "100.5 这一档变成 2.0", "99.0 这一档删除"
3. 客户端在本地快照上应用增量，维护出完整盘口
4. 用序列号(sequence)保证不丢包、不乱序；丢了就重新拉快照
```

> 💡 增量推送是行情系统省带宽的关键。序列号机制保证客户端盘口和服务器一致。

## 三、数据来源与计算

```
撮合引擎 ──成交流(trade stream)──▶ 行情服务
                                    │
                                    ├─▶ 实时维护盘口(order book)
                                    ├─▶ 聚合成 K线(按时间窗口)
                                    ├─▶ 计算 24h 统计(滑动窗口)
                                    └─▶ 推送给用户
```

- 所有行情都从撮合引擎的**成交流和盘口变化**派生计算。
- K线、统计是对成交流做**时间窗口聚合**（常用流式计算）。

## 四、多级缓存与分发

为应对海量读取，行情系统大量用缓存和就近分发：

```
撮合 ──▶ 行情核心 ──▶ Redis/内存缓存 ──▶ 推送网关集群 ──▶ CDN/边缘节点 ──▶ 用户
                        (热数据)          (扇出广播)        (就近，降延迟)
```

- **缓存**：最新盘口、K线、统计放内存/Redis，避免重复计算。
- **扇出（Fan-out）**：一份更新复制给海量连接，用网关集群分担。
- **边缘节点**：把推送节点部署在离用户近的地方，降低延迟（全球用户尤其重要）。

## 五、合约特有：指数价与标记价的计算

（与 [3.5 风控](./3.5-清算与风控.md) 呼应）

- **指数价**：实时从多家现货交易所拉价格，加权平均。要处理某个数据源异常/掉线的容错（剔除异常值）。
- **标记价**：指数价 + 基差，平滑处理，避免抖动导致误强平。
- 这两个价格的**稳定和防操纵**直接关系到强平的公平性，是行情系统的高敏感模块。

## 六、行情系统的「可信」问题 💡

- 用户看到的盘口和成交，是交易所**单方面推送**的，用户无法独立验证真假。
- ⚠️ 这给了不良交易所**造假空间**：伪造深度（假挂单）、刷成交（wash trading）、甚至画 K 线。
- 这也是去中心化（链上可验证行情）的一个动机。对交易所而言，行情的真实与稳定是信誉基础。

## 七、给量化/机构的行情服务

- **高频行情 API**：低延迟的 WebSocket/FIX 接口，给量化机构。
- **历史数据**：K线、逐笔成交的历史下载，供回测。
- **托管/Colocation**：顶级交易所给机构提供「机房就近部署」降低延迟（和传统交易所一样）。

## 小结

- 行情系统是「一对百万」的实时广播，核心挑战是高并发 + 低延迟。
- 推送用 **WebSocket 长连接 + 增量推送 + 序列号**（省带宽、保一致）。
- 行情从撮合成交流派生，K线/统计靠时间窗口聚合。
- 用多级缓存 + 扇出 + 边缘节点应对海量分发。
- 指数价/标记价的防操纵是高敏感模块；行情可信度是 CEX 的信誉软肋。
