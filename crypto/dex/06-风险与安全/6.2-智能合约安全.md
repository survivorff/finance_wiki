# 6.2 智能合约安全

> 深度：🔴 硬核
> 「信任代码」的前提是代码没 bug。但代码总会有 bug——DeFi 被黑损失累计达数十亿美元。这篇讲常见漏洞和防御。

## 为什么智能合约安全如此致命

```
智能合约的特点放大了 bug 的危害:
  1. 直接管钱: 合约里锁着真金白银, bug = 直接损失
  2. 公开可见: 代码公开, 黑客可以慢慢研究找漏洞
  3. 不可篡改: 部署后(通常)改不了, 发现 bug 也难补
  4. 可组合: 一个协议的 bug 可能波及依赖它的其他协议
  5. 不可逆: 被黑后资金转走, 链上不可逆, 极难追回
```

> ⚠️ 一句话：**在智能合约里，一个小 bug 可能在几秒内被掏空整个协议，且无法挽回。**

## 常见漏洞类型 ⭐

### 1. 重入攻击（Reentrancy）⭐ 最经典
```
原理: 合约 A 在"更新自己状态前"先给外部转账
     → 外部(恶意合约)在收到转账的回调里, 再次调用 A 提款
     → 因为 A 还没更新余额, 被反复提款掏空
著名: The DAO 事件(2016), 导致以太坊硬分叉(ETH/ETC 分裂)
防御: "检查-生效-交互(Checks-Effects-Interactions)"模式; 重入锁
```

### 2. 预言机操纵 / 闪电贷攻击
```
用闪电贷操纵价格预言机, 套取协议资金(见 3.4)
→ DeFi 被黑最常见根因之一
防御: 用 TWAP / 去中心化预言机, 不用单池瞬时价
```

### 3. 整数溢出/下溢
```
旧版 Solidity 里数字超出范围会"绕回"(如 0-1 变成最大值)
→ 可被利用造出天量代币/绕过校验
防御: 新版 Solidity 默认检查; 用 SafeMath
```

### 4. 访问控制错误
```
关键函数(提款/升级/铸币)忘了加权限校验 → 任何人都能调用
防御: 严格的权限修饰符、多签控制敏感操作
```

### 5. 逻辑/经济模型错误
```
代码没 bug, 但经济机制设计有缺陷(激励错配、可被套利掏空)
→ 比代码 bug 更隐蔽, 审计也难发现
```

### 6. 其他
- 闪电贷 + 治理攻击（借票通过恶意提案）、签名重放、价格舍入误差累积、跨合约调用假设错误等。

## 防御体系 ⭐

```
1. 代码审计(Audit): 请专业安全公司(CertiK, Trail of Bits, OpenZeppelin, 慢雾等)审查
   ⚠️ 但审计≠绝对安全! 审计过的协议仍可能被黑(审计有盲区)
2. 形式化验证: 用数学证明关键属性成立
3. 漏洞赏金(Bug Bounty): 重金悬赏白帽找漏洞(Immunefi 平台)
4. 时间锁 + 多签: 敏感操作延迟生效 + 多人批准(见 3.2)
5. 渐进部署: 限额上线, 逐步放开(限制早期 bug 的损失)
6. 监控 + 应急: 实时监控异常, 有暂停(pause)机制和应急响应
7. 保险: DeFi 保险(如 Nexus Mutual)对冲被黑风险
```

## 经典攻击复盘（教训）

| 事件类型 | 教训 |
| --- | --- |
| The DAO（2016，重入） | 催生「检查-生效-交互」规范；引发 ETH/ETC 分叉 |
| 各类闪电贷预言机攻击 | 永远不用单池瞬时价做预言机 |
| 跨链桥被黑（多起） | 桥是最高危环节，验证逻辑+多签是命门（见 [3.5](../03-技术架构/3.5-跨链与桥.md)） |
| 治理攻击 | 借票/买票通过恶意提案，需治理防护 |
| 私钥/多签泄露 | 再好的合约也扛不住密钥管理失误（同 [CEX Bybit](../../cex/06-风险与监管/6.4-经典案例复盘.md)） |

## 「代码即法律」的两难 💡

```
理想: Code is Law — 代码怎么写就怎么执行, 无人能干预
现实困境: 当 bug 被利用(比如 The DAO):
  - 严格派: 代码就是法律, 黑客"合法"利用了规则, 不该回滚
  - 务实派: 这是被盗, 应该回滚救用户 → 以太坊最终硬分叉回滚(ETC 坚持不回滚)
```

> 💡 这是 DeFi 最深刻的哲学拷问：**「代码即法律」在理想和现实间的张力**。
> 完全的「不可篡改」很纯粹，但当几亿美元被一个 bug 卷走时，社区往往难以坐视。
> 现实中，越来越多协议保留了「紧急暂停」「可升级」等中心化后门——这又削弱了去中心化。**纯粹性与务实性的永恒矛盾。**

## 小结

- 智能合约管钱、公开、不可篡改、可组合、不可逆——这些特点让 bug 极其致命。
- 常见漏洞：重入（The DAO）、预言机/闪电贷操纵、整数溢出、访问控制、经济模型错误。
- 防御：审计（但≠绝对安全）、形式化验证、漏洞赏金、时间锁+多签、渐进部署、监控、保险。
- 「代码即法律」在理想与现实间张力巨大（The DAO 硬分叉），导致很多协议保留中心化后门。
