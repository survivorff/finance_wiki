## 一、订单簿的数据结构选型

撮合的核心操作有四个，数据结构必须让它们都快：
1. **找最优价**（best bid / best ask）—— 撮合时每笔都要。
2. **同价位按时间排队**（FIFO）—— 时间优先。
3. **插入新挂单**。
4. **撤单**（按 order_id 快速定位删除）。

### 方案对比

| 方案 | 找最优价 | 插入 | 撤单 | 适用 |
| --- | --- | --- | --- | --- |
| 排序数组 | O(1) | O(n) | O(n) | 不实用 |
| 平衡树(红黑树/TreeMap) | O(log n) | O(log n) | O(log n) | 通用、价位稀疏 |
| 跳表(SkipList) | O(log n) | O(log n) | O(log n) | 通用 |
| **数组+价格索引(price level array)** | **O(1)** | **O(1)** | **O(1)** | 价位密集、有 tick size（主流高性能选择）|

### 主流高性能设计：价格档位数组
利用「价格是离散的」（有最小变动单位 tick size）这个事实：

```
把价格映射成整数索引:  price_index = (price - min_price) / tick_size
用一个数组 levels[price_index] 直接定位到该价位的订单队列
维护 best_bid_idx / best_ask_idx 两个指针，撮合后增减指针找下一个非空档位
```

- 每个价格档位 = 一个**双向链表/队列**（保证 FIFO + O(1) 头部出队）。
- 一个 `HashMap<order_id, node_pointer>` 用于 O(1) 撤单定位。

```
                price level array (按价格索引)
   idx:   ...  [99.5]   [100.0]   [100.5]  ...
                  │         │         │
                  ▼         ▼         ▼
              [订单队列]  [订单队列]  [订单队列]   ← 每档是 FIFO 双向链表
                            ▲
                  best_bid_idx 指针

   order_map:  { order_id_123 → 指向某个链表节点 }   ← O(1) 撤单
```

## 二、核心数据结构（伪代码）

```python
class Order:
    id: int
    side: BUY | SELL
    price: int          # 用整数(最小单位)表示，避免浮点误差！
    quantity: int       # 同样用整数
    timestamp: int
    user_id: int

class PriceLevel:
    price: int
    orders: Deque[Order]      # FIFO 队列
    total_qty: int            # 该档位总量(用于深度展示)

class OrderBook:
    bids: PriceLevelArray     # 买盘，按价格组织
    asks: PriceLevelArray     # 卖盘
    order_map: Dict[int, (PriceLevel, Order)]   # O(1) 撤单定位
    best_bid: int             # 当前最优买价索引
    best_ask: int             # 当前最优卖价索引
```

> ⚠️ **铁律：金额和价格一律用整数（最小单位）存储和计算，绝不用浮点数。**
> 浮点数有精度误差（0.1 + 0.2 != 0.3），在金融系统里是灾难。用「聪/satoshi」式的最小整数单位。

## 三、撮合主逻辑（限价单，伪代码）

```python
def match_limit_order(book, incoming: Order):
    trades = []
    if incoming.side == BUY:
        # 只要还有量，且对手盘(卖盘)最优价 <= 我的限价，就成交
        while incoming.quantity > 0 and book.best_ask exists \
              and book.asks[book.best_ask].price <= incoming.price:
            level = book.asks[book.best_ask]
            resting = level.orders.front()          # 时间优先：队首先成交
            fill = min(incoming.quantity, resting.quantity)

            trades.append(Trade(
                price = resting.price,               # 成交价 = 挂单价(maker 定价)
                qty   = fill,
                taker = incoming, maker = resting))

            incoming.quantity -= fill
            resting.quantity  -= fill
            if resting.quantity == 0:
                level.orders.pop_front()             # 该挂单吃完，出队
                book.order_map.remove(resting.id)
            if level.is_empty():
                advance(book.best_ask)               # 移动最优价指针到下一档

    # SELL 侧对称处理 ...

    if incoming.quantity > 0:
        book.insert(incoming)                        # 剩余量挂入盘口，成为 maker
        book.order_map[incoming.id] = ...
    return trades
```

### 几个关键细节
- **成交价取 maker（挂单方）的价格**，不是 taker 的价格。这是「价格优先」的体现，也对 taker 有利（不会比限价更差）。
- **市价单**：去掉 `price <= incoming.price` 的判断，一直吃到量满或盘口空。要加「最大滑点保护 / 可用资金保护」防止吃穿。
- **自成交防护（Self-Trade Prevention, STP）**：撮合到同一用户的对手单时，按策略处理（取消 maker / 取消 taker / 都取消），防止刷量和洗售。

## 四、定序器（Sequencer）—— 单线程的入口

撮合引擎单线程，但请求来自多个网关。需要一个**定序器**把并发请求变成一个**全局有序的指令流**：

```
多个网关 ──▶ [定序器 Sequencer] ──▶ 单一有序指令流 ──▶ 撮合引擎(单线程消费)
            分配全局递增 seq_id      (这就是"唯一真相序列")
            并持久化到 WAL/Kafka
```

- 定序器给每条指令分配**全局唯一递增序号**，并**先持久化再投递**。
- 这条「指令序列」就是系统的**唯一真相（source of truth）**。撮合引擎是它的纯函数消费者。

## 五、快照 + 重放（恢复机制）

```
正常运行:  指令流 ──▶ 撮合引擎(内存) ──▶ 成交输出
           每隔 N 条/T 秒，把内存订单簿全量写一份 snapshot(含最后处理的 seq_id)

崩溃恢复:
  1. 加载最近一次 snapshot (恢复到 seq_id = S)
  2. 从 WAL 重放 seq_id > S 的所有指令
  3. 因为是确定性状态机，重放后状态 == 崩溃前状态
```

> 💡 **确定性是一切的根基**：撮合逻辑里**不能有任何不确定来源**——
> 不能用 `now()`（时间戳由定序器赋予并记入日志）、不能用随机数、不能依赖外部服务、不能有并发竞态。
> 否则重放就会偏离，一致性崩塌。

## 六、高可用：状态机复制（再深一层）

```
            定序器(主) ──持久化指令流──┐
                                      ├──▶ 撮合引擎 主(对外服务)
   指令流(同一份 seq) ─────────────────┼──▶ 撮合引擎 备1(热备, 消费同一流)
                                      └──▶ 撮合引擎 备2(热备)
   主挂掉 → 备已消费到相同 seq → 状态完全一致 → 秒级接管
```

- 关键：备机消费**完全相同的指令流**，所以状态严格一致。
- 切换时只需确认备机已追平到主机最后的 seq_id。
- 这套思想 = Raft/Paxos 式「复制状态机」在交易撮合上的应用。

## 小结

- 订单簿主流用「价格档位数组 + 每档 FIFO 队列 + order_map」实现 O(1) 关键操作。
- 金额价格一律用整数，杜绝浮点误差。
- 撮合取 maker 价成交，需做自成交防护。
- 定序器把并发请求变成「全局有序、先持久化」的唯一真相指令流。
- 确定性状态机 + 快照重放 = 可恢复；状态机复制 = 高可用。
